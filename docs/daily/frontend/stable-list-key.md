# 列表渲染时保持稳定 Key

## 适用场景

列表会插入、删除、排序时，如果直接用索引当 key，组件状态和 DOM 复用很容易错位。

## 推荐做法

- 优先使用后端返回的稳定 id
- 没有 id 时，在进入前端数据层时就生成稳定标识
- 不要在渲染时临时 `Math.random()`

## 常见问题

- 表单输入框值错位
- 过渡动画异常
- 子组件状态被错误复用

## 经验结论

- `key` 的本质不是消除警告，而是帮助框架稳定识别节点身份
- 只要列表顺序可能变化，就尽量不要用索引当 key
