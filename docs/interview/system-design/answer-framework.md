# 设计题回答框架

## 推荐结构

1. 明确需求和约束
2. 给出核心模块划分
3. 说明数据流与关键接口
4. 分析容量、瓶颈与扩展方案
5. 补充监控、容灾和安全

## 常见失分点

- 一上来就谈技术，不先确认需求
- 只讲正常流程，不讲异常和扩容
- 图画得很大，但没有落到关键瓶颈

## 推荐答题句式

- “我先确认一下需求和约束，再给一个核心方案。”
- “这里我会重点关注容量瓶颈、数据一致性和可扩展性。”
- “如果流量继续上涨，我会优先从这几个点扩容。”

## 题库示例

### 问题 1：设计一个短链接系统，你会怎么回答

**回答要点**

- 先确认核心目标：生成短链、跳转稳定、访问统计、容量预估
- 再拆模块：生成服务、跳转服务、存储、缓存、统计
- 再考虑热点短链、高并发跳转、缓存和容灾
- 最后补监控、限流和恶意链接治理

**常见追问**

- 短码冲突怎么处理
- 热门短链访问量暴涨时怎么抗

### 问题 2：设计题最容易失分的地方是什么

**回答要点**

- 没先确认需求和边界条件
- 只讲主流程，不讲异常和扩展
- 只堆技术名词，没有结合实际瓶颈

**常见追问**

- 如果时间不够，哪些点一定要讲到
- 怎么让回答显得更有条理
